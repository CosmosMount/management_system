# 业务总览与共享框架

## 系统定位

项目管理模块服务于团队内部项目推进。系统的目标不是替代飞书、文档或线下沟通，而是把项目、阶段、任务、责任人、证据和待办人沉淀在同一个框架里。

系统坚持四个原则：

1. 业务对象必须有明确状态。
2. 状态推进必须由有权限的人完成。
3. 关键动作必须留下可追踪记录或证据。
4. 外部通知只辅助触达，不决定业务是否成立。

## 管理对象

项目管理模块负责回答：

- 哪个项目正在推进，归属哪个车组和技术组。
- 项目当前是未开始、进行中、已完成还是已取消。
- 项目阶段是否按顺序完成提交与审批。
- 项目下有哪些任务，分别由谁负责。
- 任务何时截止，是否逾期，是否完成交付验收。
- 周报、交付文档、视频链接和验收记录是否可追踪。
- 哪个处理人应继续推动当前待办。

## 共享框架

### 账号

用户通过飞书身份进入系统。系统在本地保留用户姓名、头像、飞书身份标识和必要的业务资料。业务权限不由飞书组织关系自动决定，而由系统内的角色配置决定。

### 角色

项目管理动作由超级管理员、项管、项目创建人、车组组长、技术组组长和任务负责人共同完成。角色只决定用户可能具备的职责，具体动作仍要结合项目范围和当前状态判断。

### 范围

多数角色都带有业务范围：

- 车组范围：角色只对指定车组生效。
- 技术组范围：角色只对指定技术组生效。
- 全局范围：角色不绑定具体车组或技术组。

一个业务对象通常同时带有车组和技术组。系统根据对象范围和用户角色范围共同判断权限。

### 状态机

项目、阶段和任务都通过状态表达业务进度。状态不是展示标签，而是权限、通知、统计和下一步动作的依据。

状态机的基本要求是：

- 只允许进入明确的下一状态。
- 终态对象不能继续常规推进。
- 退回、驳回、归档等动作必须有清晰业务语义。
- 不能为了界面方便绕过状态条件。

### 证据

项目管理使用飞书文档链接、视频链接、周报和验收记录作为证据。系统记录证据与业务对象的关系，但不替代人工对证据内容的判断。

### 通知

通知用于把待办推送给处理人。通知可以失败，业务状态仍以系统内记录为准。处理人应以系统页面中的当前状态和可执行动作作为最终依据。

### 定时提醒

定时提醒用于发现活跃任务、逾期任务和周报待办。提醒只改变触达，不自动改变项目、阶段或任务状态。

## 扩展视角

未来扩展项目管理能力时，不应重新设计一套账号、角色和通知体系。合理做法是：

- 复用用户、角色、范围和通知能力。
- 为新增能力定义清晰对象、状态、动作和证据。
- 明确新增能力与项目、阶段、任务之间的数据关系。
- 保持业务判断在系统内完成，外部系统只承载身份、文档、消息或附件。
