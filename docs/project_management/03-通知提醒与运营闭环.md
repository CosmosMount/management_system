# 通知提醒与运营闭环

## 基本原则

通知是项目状态变化后的触达手段，不是项目状态本身。系统遵循以下原则：

1. 先保存业务状态，再发送通知。
2. 通知失败不回滚业务动作。
3. 通知接收人由项目范围、全部任务负责人和角色计算得出。
4. 飞书卡片只提供入口和摘要，最终处理依据是系统中的当前状态。

## 通知渠道

当前主要使用飞书私信通知具体处理人，例如项目管理者、全部任务负责人和验收人。

用户需要已经存在于系统中，且飞书应用具备对应消息能力，才能稳定收到私信。

## 项目通知

项目通知围绕项目创建、启动、完成和取消发送。

| 场景 | 接收人 |
|---|---|
| 新项目创建 | 匹配车组组长、匹配技术组组长、项管 |
| 项目启动、完成、取消 | 匹配车组组长、匹配技术组组长、项管 |

项目通知用于让管理角色了解项目大生命周期变化。通知本身不改变项目状态。

## 阶段通知

| 场景 | 接收人 |
|---|---|
| 阶段提交后待审批 | 项目负责人、提交人、匹配车组组长、匹配技术组组长、项管 |
| 阶段审批通过或驳回 | 阶段负责人 |

## 任务通知

任务通知围绕任务分配、交付、验收和风险发送。

| 场景 | 接收人 |
|---|---|
| 新任务指派 | 全部任务负责人 |
| 任务提交后待验收 | 匹配车组组长、匹配技术组组长、项管 |
| 任务验收通过 | 全部任务负责人 |
| 任务逾期 | 全部任务负责人，以及匹配车组组长、匹配技术组组长、项管 |
| 任务风险同步 | 全部任务负责人、项目负责人、匹配车组组长、匹配技术组组长、项管 |
| 周报提醒 | 活跃任务的全部负责人 |
| 每日任务提醒 | 全部任务负责人、项目负责人、匹配车组组长、匹配技术组组长、项管 |

任务待验收通知可以包含交付文档链接。验收人仍需进入系统完成通过或驳回。

## 定时运营规则

定时任务用于固定节奏触发提醒和汇总：

- 每日早上执行进度逾期检查。
- 每日早上提醒活跃任务的全部负责人和管理角色。
- 每周一早上提醒活跃任务的全部负责人提交周报。

定时提醒不会自动完成、取消、驳回或归档任何项目、阶段或任务。它只帮助团队发现待办、积压和风险。

## 接收人计算

通知接收人不应手工写死，而应从以下信息计算：

- 项目的车组和技术组。
- 当前状态对应的处理角色。
- 用户当前拥有的角色及其范围。
- 特定对象负责人，例如任务负责人、项目负责人、阶段负责人。

如果某个角色没有配置人员，业务状态仍会推进，但对应通知可能无人接收。角色配置缺失应作为运营问题处理。

## 失败处理

通知失败的常见原因包括：

- 用户尚未登录过系统或未完成通讯录同步。
- 角色配置缺失或范围不匹配。
- 飞书应用权限或机器人会话不满足发送条件。
- 外部服务临时不可用。

通知失败后，项目、阶段和任务状态以系统内记录为准。处理人可以通过系统页面继续处理，不应要求回滚已经完成的提交、验收或状态变更。

## 运营闭环

通知和提醒的目标是形成闭环：

1. 状态变化产生待办。
2. 系统通知对应处理人。
3. 处理人在系统中完成动作。
4. 状态进入下一环节或终态。
5. 长时间停留或逾期的对象被定时提醒再次暴露。

当待办长期停留时，应优先检查角色配置、处理人是否在系统中、交付证据是否齐全，以及当前状态是否符合预期。
