# 身份角色与权限边界

## 身份来源

系统用户来自飞书登录或飞书通讯录同步。飞书负责确认“这个人是谁”，系统负责判断“这个人在项目管理里能做什么”。

用户首次进入系统后，会在本地形成用户记录。管理员也可以通过通讯录同步提前录入用户，便于在对方登录前分配角色。

## 角色配置原则

角色由超级管理员配置。系统不会因为用户属于某个飞书部门、群聊或组织自动授予业务权限。

同一用户可以拥有多个角色。例如一个人可以同时管理某个车组的项目，也可以作为某个具体任务的负责人。

## 当前角色

| 角色 | 范围 | 主要职责 |
|---|---|---|
| 超级管理员 | 全局 | 管理角色配置，拥有项目管理兜底权限 |
| 项管 | 全局 | 创建项目、参与阶段兜底审批、任务验收和风险介入 |
| 车组组长 | 指定车组 | 管理和验收本车组范围内的项目对象 |
| 技术组组长 | 指定技术组 | 管理和验收本技术组范围内的项目对象 |
| 项目创建人 | 单个项目 | 管理自己创建的项目和挂载任务 |
| 任务负责人 | 单个任务，可多人 | 开始任务、提交周报、提交交付 |

## 权限判断模型

权限由三类信息共同决定：

1. 用户拥有哪些角色。
2. 角色是否匹配业务对象的车组或技术组。
3. 当前业务状态是否允许执行该动作。

有角色不代表任何状态下都能操作。例如任务负责人只有在自己参与负责的任务上才能提交周报和交付；项目完成必须先满足全部阶段完成条件。

## 进度管理权限

进度管理围绕项目、阶段和任务推进，主要权限如下：

| 动作 | 处理人 |
|---|---|
| 创建项目 | 超级管理员、项管、任意车组组长、任意技术组组长 |
| 启动、完成、取消项目 | 项目负责人，超级管理员兜底 |
| 创建任务 | 可管理该项目的人 |
| 开始任务 | 任一任务负责人 |
| 提交周报 | 任一任务负责人 |
| 提交任务交付 | 任一任务负责人 |
| 验收任务交付 | 超级管理员、项管、匹配车组组长、匹配技术组组长 |
| 提交阶段材料 | 阶段负责人，超级管理员兜底 |
| 审批阶段材料 | 项目负责人；自审关闭时由项管、超级管理员或匹配组长兜底 |
| 同步任务风险 | 任一任务负责人 |
| 归档任务 | 可管理该项目的人 |
| 完成项目 | 项目负责人，且全部阶段已完成 |

## 超级管理员边界

超级管理员用于配置角色和处理特殊运营问题，不应替代常规流程。日常项目推进、任务执行和验收仍应由对应业务角色完成。

使用超级管理员处理异常时，应遵守两个原则：

- 不跳过必要证据。
- 不把一次性处理方式写成常规流程。

## 用户变更

用户离开团队、角色调整或职责变化时，应调整系统内角色配置。历史业务记录应保留原处理人姓名和当时的业务状态，不因人员变化而删除或重写。
